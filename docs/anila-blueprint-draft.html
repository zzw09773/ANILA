TL;DR · 一頁摘要

一句話：院內自己的 AI Agent 作業系統

ANILA 是一套部署於院內、以落地 LLM 為核心的多 Agent 平台。 它統一「模型管理、API Key 治理、Agent 註冊審核、對話記錄與審計」， 對外以 OpenAI 相容介面供給各單位呼叫，對內讓開發者基於官方樣板快速建立自己的 agent。

不變條件 01
流量全數落地

所有 LLM 呼叫進自家 on-prem endpoint；平台本身不含雲端 fallback，資料不出院。

不變條件 02
敏感對話單向閂鎖

任一 agent 標記 classified，整段對話在 CSP + Router + UI 三層永久鎖定加密模式，無降級路徑。

不變條件 03
三層認證清楚切分

管理面 JWT、資料面 API Key、服務間 Service Token，憑證層級不得越界共用。

起點 · Motivation

為什麼現在要做 ANILA

需求已浮現

院內多個單位已自行嘗試接 LLM API、自製 agent。 若不收斂，憑證散落、用量不可追蹤、敏感資料外流風險升高。

基礎散裝但未整合

現已有三個子專案（onyx、myCSPPlatform、AgenticRAG）合併進同一 repo， 但各自獨立運作；缺一個統一的 runtime 入口與治理面。

落地 LLM 已具備

院內已能部署 vLLM / Ollama / TGI 類 OpenAI 相容 endpoint。 ANILA 只要把它們接進來，即可把推論與資料完全留在院內。

合規需求提前

個資、病歷、簽核文件一旦進入 agent 對話，必須能審計、能閂鎖、能追蹤用量。 這不是可選項，是做 AI 平台的入場門票。

核心主張 · Invariants

四條貫穿全平台的不變條件

設計決策的地基；任何功能提案都須通過這四條檢驗

INVARIANT · CSP as single chokepoint
所有 LLM / Agent 呼叫一律經 CSP 代理

Router 呼叫主 LLM、Router 分派 agent、agent 內部再打 LLM — 全部走 CSP /v1/chat/completions。 Usage 在 CSP 一次結算，Router 與 agent 不持 upstream key。

INVARIANT · on-prem runtime-first
無雲端 fallback、無 token quota

LLM 推論由各單位內部基礎設施承擔。Wave 0 已完整移除 rate-limit / quota 子系統； 平台不做 token 或 request 級節流，DDoS 緩衝留給 nginx IP-level limit_req。

INVARIANT · one-way latch
Classified 對話三層閂鎖，不可降級

只要 agent 標記 requires_encryption=true，CSP + Router + UI 同步鎖定該對話為加密模式。 UI 側沒有任何降級路徑。

INVARIANT · three-tier credential
管理面 / 資料面 / 服務間三組憑證嚴格分離

JWT管人、sk-...管 runtime、Service Token管 s2s。 Downstream agent 不驗 user JWT、不驗 user API key，只信 CSP 注入的 identity header。

資料流 · One chat lifecycle

一次對話請求的七個動作

1. 使用者於 UI 送訊息
     │
2. UI POST /v1/chat/completions  model="anila-router"   → Router
     │
3. Router 用 caller 的 Bearer API Key 呼叫 CSP：
     • GET  /v1/agents            取 agent manifest
     • POST /v1/chat/completions  主 LLM 判斷是否分派
     │
4. 主 LLM 回「我需要叫 agent X」
     Router 以 caller 的 API Key 轉發到 agent X 的 endpoint
     agent X 內部可再呼叫 CSP /v1/* 拿 RAG、Embedding
     │
5. Router / CSP 將 SSE 逐 chunk forward 回 UI
     若 agent requires_encryption，meta 標注 classified=true
     │
6. UI 收到 classified=true → 對話永久閂鎖為加密模式
     │
7. token_usage 於 CSP 逐層寫入：主 LLM 1 筆、agent 1 筆、agent 內部 LLM 1 筆
     三筆皆正確歸屬到 caller API Key 的擁有者

驗收訊號：一次 smoke query 後，資料庫 token_usage 應出現三筆可追蹤紀錄、 SSE 為逐 chunk 串回而非一次性 dump，且若該 agent 為 classified，UI 對話框狀態應永不降級。

關鍵設計決策 · Design Decisions

每條決策都有「為什麼」與「邊界」

決策理由邊界條件
CSP 即資料面單點 Usage 可信度、憑證收斂、合規審計都要一個統一 chokepoint Router / agent 只持 CSP 發的 API Key，絕不直連 upstream
Registry 拆雙表 model_registry 管 LLM，agents 管 endpoint agent — 治理責任不同 agents.base_model_id FK 保留關聯，不強合併
新增 developer role 開發者需註冊 agent、下載樣板、查審核狀態 — 權責與 user 不同 Agent 須 admin 批准後才能被其他 user 發現或呼叫
Router 不內建 RAG Router 是 orchestrator，不是 knowledge base；檢索只在被分派到的 agent 內發生 Router 不預設先跑文件檢索、不內建單位知識庫
Onyx 暫列 future module 相關單位 API 尚未就緒；先定角色邊界避免與主線競爭 僅交付 docs/onyx-application-plan.md，不排入關鍵路徑
不做 token / request quota 推論由院內基礎設施承擔，平台不重複節流 nginx IP-level limit_req 保留作 DDoS 緩衝
範圍邊界 · Scope

ANILA 做什麼、不做什麼

明確劃線 — 避免「平台什麼都要做」的蔓延

ANILA DOES
這些是 ANILA 的責任範圍
  • 統一管理落地模型與 API Key 發放
  • 以 OpenAI 相容介面對外供給推論
  • 提供 agent 開發樣板 + 註冊審核流程
  • 統一 UI：對話、分享、交接、審計
  • 敏感對話單向閂鎖（one-way latch）
  • 三層憑證治理與 audit log
  • 服務間注入 trusted identity headers
ANILA DOES NOT
這些明確排除 — 不要被要求做
  • 雲端 LLM fallback（無論什麼藉口）
  • Token / request 級節流與計費
  • Router 內建 RAG 或文件預檢索
  • 主循環內嵌業務知識庫
  • 以 LLM 判定做為最終權限授權依據
  • 為每個 agent 重寫一套 runtime 底座
  • 把控制面與 runtime UI 合而為一

驗收標準 · Verification

Phase 3 結束時的可觀察訊號

每條訊號都對應一個實際可跑的命令或資料庫查詢

Smoke Path · direct answer
未註冊任何 agent 時

Router 走 direct-answer 路徑，不依賴預設 RAG； 主 LLM 呼叫經 CSP，token_usage 新增 1 筆 LLM-level 紀錄。

Smoke Path · dispatch
已註冊且被分派時

主 LLM 決策 → 呼叫 dispatch_to_agent()； dispatch 經 CSP proxy 轉給 agent；agent 內部 LLM 呼叫也經 CSP。

通過 Phase 3 的必要檢查項

  • Router 啟動能從 CSP 拿到該 user 可用 agent 清單（RemoteAgentRegistry works）
  • 三筆 token_usage 全部正確歸屬到 test user（LLM 1 筆 + agent 1 筆 + agent 內部 LLM 1 筆）
  • SSE 為逐 chunk 串回 curl 終端，非一次性 dump
  • Router /health 暴露 cached_agents 與 last_refresh_error（為 null 代表正常）
  • Developer 可於 CSP 前端看到自己的 agent、下載官方 template、查看審核狀態
  • Admin 管理操作（建立 / 停用 / 改密碼 / 刪 agent）全數寫入 audit_logs
風險與相依 · Risks & Dependencies

目前尚未解決的四件事

外部相依
相關單位 API 尚未就緒

Onyx 所需的跨系統 workflow / compliance API（請假、簽核、其他內部系統）未備齊。 決策：Phase 4 僅出文件，不進開發排程，待 API 就緒再重新啟動。

設計相依
ANILA UI 待 Claude Design 產出規格

Runtime UI wireframe / visual / interaction 稿由 Claude Design 負責，工程實作再依稿接 Router / CSP。 Phase 1–3 不被 UI 阻塞，以 curl + OpenWebUI 先驗證。

基礎設施
落地 LLM 選型與 baseline 未定

vLLM / Ollama / TGI / llama.cpp 皆為 OpenAI-相容候選；需依院內 GPU 規模與模型需求選定。 效能 baseline 與選型請主管核定採購規格。

技術債
Alembic migration 尚未取代 startup backfill

MVP 階段沿用 CSP 現有 startup schema/backfill 機制加速開發。 Phase 7 會改為正式 Alembic migration，避免長期維運風險。
